# KirinNet 安全模型 v1.0

## 概述

KirinNet 的安全模型从根本上有别于传统互联网。传统互联网采用"平台中心化安全"：平台
承担用户账号安全的责任，通过防火墙、密码哈希、多因素认证等手段构建纵深防御。一个平台
被攻破，所有用户的密码哈希泄露，撞库连锁灾害随之而来。

KirinNet 将安全责任归还给用户——如同物理世界中，你对自己的钱包、家门钥匙和身份证负责，
而不是把它们交给商场保管。用户节点被攻破，影响仅限于该节点一个。域名即是账号，域名即
是身份，不存在"平台替你保管密码"的概念。

---

## 一、三层信任模型

KirinNet 的信任体系分为三层，每层解决不同的信任需求：

```
┌─────────────────────────────────────────────────────────────┐
│  第三层：物理层 — 域名 = 身份证 = 真实个人                      │
│  机制：政府CA认证                                              │
│  解决：监管合规、物理追责、陌生人最低信任锚                       │
├─────────────────────────────────────────────────────────────┤
│  第二层：公共层 — 陌生人之间的最低信任                           │
│  机制：政府CA签发的域名证书                                      │
│  解决：证明"该域名背后是一个真实的人"，但不暗示"可信"              │
├─────────────────────────────────────────────────────────────┤
│  第一层：好友层 — 最强信任，自验证网络                            │
│  机制：好友间RSA密钥对，首次交换后锁定                             │
│  解决：身份冒充、中间人攻击、信任传导                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.1 好友层：RSA自验证信任网络

这是 KirinNet 安全模型的核心创新。

**原理：**

每个用户节点生成一对RSA密钥。当两个用户成为"好友"时，双方互相存储对方的公钥。
此后，所有通信必须用该公钥验证。公钥一旦存储即锁定，未经旧密钥签名不得更换。

**为什么中间人攻击在此模型中失效：**

传统网络中，中间人攻击可以在TLS握手阶段替换证书，因为客户端没有该网站的"先验
知识"。但在 KirinNet 中：

- 好友公钥不是在通信信道上获取的，而是在"加好友"这个动作中带外交换的
- 带外信道包括：线下见面扫码、已有好友的信任介绍、语音通话中念指纹
- 中间人不在这个带外信道中，因此无法注入伪造公钥

此后所有通信，任何中间人尝试替换证书的行为都会导致RSA签名验证失败。好友会立即
发现"对方的密钥不匹配"，进而触发告警。

**分布式入侵检测：**

这是传统PKI体系不具备的能力。假设Alice的节点被物理攻破，攻击者替换了她的RSA密钥：

1. Alice有50个好友
2. 所有50个好友存储的都是Alice的旧公钥
3. 攻击者用新密钥给任何好友发消息 → 所有50人同时发现密钥不匹配
4. 不需要中心化告警服务器。好友之间通过线下渠道（电话、当面）互相确认：
   "Alice的密钥是不是变了？"
5. 攻击者需要同时攻陷Alice的所有50个好友才能完成冒充。这在实际中不可能。

这本质上是一种**社交共识安全模型**——安全不依赖于任何第三方，而是依赖于你的
社交关系网络。

### 1.2 公共层：政府CA的职责与边界

政府CA的作用是解决一个精确的问题：**证明"域名 = 真实身份"的映射关系**。

CA**不做**的事情：
- CA不推荐信任。浏览器显示"该域名已通过XX国政府实名认证"，但不暗示该网站"安全"
  或"可信"
- CA不担保行为。实名认证的人仍然可以是骗子

CA**仅做**的事情：
- 验证域名拥有者是一个真实的人（或实体）
- 提供一个从域名到物理身份的追溯通道
- 使监管和执法成为可能

**这就像护照：** 中国护照证明你是中国公民，但伊拉克边境是否放行由伊拉克自己决定。
同理，中国政府CA签发的证书证明该域名背后是一个经过实名认证的中国人，但伊朗用户
是否信任这个证书——完全取决于伊朗用户自己。

**跨国信任：决定权在自身：**

- 每个国家的CA是独立的信任锚
- 浏览器可以配置信任哪些国家的CA
- 用户可以自行决定：信任中国CA签发的证书？信任伊朗CA签发的证书？两者都信任？
- 政府CA不构成全球信任体系，只是物理真实性的一个参考信息

这比当前Web PKI体系更诚实。当前体系下，全球几百个CA中任何一个被攻破，就可以
为任意域名签发证书。KirinNet将CA边界与国家主权对齐，信任范围由用户自己划定。

### 1.3 物理层：风险模型与物理世界的对齐

在传统互联网中：

```
某个网站数据库泄露 → 1000万用户密码哈希被拖库 → 撞库 → 用户在100个网站的
账号全部沦陷 → 连锁灾难
```

在 KirinNet 中：

```
某个用户节点被物理攻破 → 仅该用户个人数据泄露 → 攻击者无法伤害该用户的
任何好友 → 攻击范围严格限于单点
```

**风险完全对应于物理世界：**

- 对一个人发起物理攻击（绑架、胁迫、盗窃设备），你可以获得这个人的所有信息
- 对一个人的KirinNet节点发起攻击，你可以获得该用户的所有数据
- 两者的难度和后果是等价的

**故障隔离：**

传统网络的安全灾难是"量变引起质变"——攻击者攻破一个服务器就获得百万用户的数据。
KirinNet将故障域严格限制在单点。攻击者需要逐个攻击100万用户节点才能获得100万
用户的数据，这在实际中不可能。

**好处：**
- 不存在"撞库"这个概念——因为每个用户的身份由自己的域名和密钥定义，不存在
  统一的用户名/密码数据库可以被撞
- 不存在"平台被黑导致用户数据泄露"——因为没有平台持有用户数据
- 不存在"某网站被拖库导致你在其他网站的账号被盗"——因为你的身份不依赖于
  任何网站


## 二、好友系统的具体机制

### 2.1 首次密钥交换

加好友时，发起方生成一个 KirinNet 好友 URI：

```
kirinnet://friend?id=<uuid>&key=<rsa_pub_hex>&nick=<name>&fp=<sha256_fingerprint>
```

**五种交换方式，信任级别递减：**

| 方式 | 信道 | 信任级别 | 适用场景 | 需要CA |
|------|------|---------|---------|--------|
| CA直连 | TLS直连（CA证书验证） | 最高（CA级） | 任意陌生人 | 双方都需要 |
| 线下扫码 | 物理世界（面对面） | 最高（物理级） | 初次认识、会议 | 不需要 |
| 好友介绍 | 已加密信道（通过已有好友转发） | 较高 | 信任传导 | 不需要 |
| 临时握手页 | 临时HTTPS页面（有时效） | 中等 | 远程即时交换 | 不需要 |
| 公开页面 | 个人主页发布好友URI | 最低 | 需后续验证 | 不需要 |

### 2.1.1 CA证书直连：零摩擦好友请求

对于持有政府CA证书的用户，加好友不需要任何带外信道。CA证书本身就是信任锚——
它证明了"该域名背后是一个经过实名认证的真实个人"。一旦TLS握手成功，双方身份
已通过CA验证，密钥交换直接在加密信道内完成。

**流程（两阶段：请求不含密钥，接受后交换）：**

```
阶段一：发起好友请求（不含公钥）

1. Alice 通过浏览器或客户端直连 Bob 节点 → TLS握手（CA证书）
2. Alice → POST https://bob.kirinnet.org:port/.kirinnet/friend/request
   Body（明文，TLS加密）:
   {
     sender_uuid: <Alice的UUID>,
     sender_domain: "alice.kirinnet.org",
     sender_nick: "Alice",
     message: "Hi Bob, 我是Alice，想加你好友",
     timestamp: <ISO8601>,
   }
   // 注意：此阶段不发送Alice的公钥。只是一个"我想加你"的意图声明。

3. Bob 节点收到请求 → 推送到Bob的界面
   → Bob看到: "Alice (alice.kirinnet.org) 请求添加你为好友"
   → Bob 可以选择：接受 / 拒绝 / 忽略

阶段二：接受后交换公钥

4. Bob 点击"接受"
5. Bob → POST https://alice.kirinnet.org:port/.kirinnet/friend/accept
   Body（用Alice的公钥加密，从TLS证书提取）:
   {
     accepter_uuid: <Bob的UUID>,
     accepter_domain: "bob.kirinnet.org",
     accepter_public_key: <Bob的RSA公钥>,   // ← 此时才暴露公钥
     accepter_nick: "Bob",
     request_timestamp: <原始请求的时间戳>,
     accept_timestamp: <ISO8601>,
   }

6. Alice 解密 → 提取Bob的公钥 → 存入好友列表
7. Alice → 返回确认（包含Alice的公钥）
   Bob → 提取Alice的公钥 → 存入好友列表
```

**为什么两阶段设计更安全：**

传统思维中，好友请求直接附带公钥。但公钥越早暴露，攻击面越大。
两阶段分离后：

| 阶段 | 暴露的信息 | 攻击者能做什么 |
|------|-----------|---------------|
| 请求 | Alice的域名和UUID | 无。没有密钥可以攻击 |
| 接受 | 双方公钥 | 仅对已确认的好友关系可见 |

- **公钥暴露窗口从"发送请求"缩短为"接受请求"** — 只有确认了对方身份后才交换密钥
- **陌生人请求同样安全** — 即使是非CA用户之间，第一阶段也只是声明意图，
  不泄露任何密钥材料
- **拒绝的请求不留下密钥足迹** — 如果Bob拒绝，Alice的公钥从未暴露给Bob

**请求-接受的时间戳绑定：**

接受包中引用了原始请求的时间戳（`request_timestamp`），这形成了一个加密的因果链：
- 接受包必须对应一个特定的、已存在的请求
- 不能将接受包重放到另一个请求（时间戳不匹配）
- Alice验证时检查：`request_timestamp` 是否与自己发出的请求匹配

**为什么不需要临时握手页：**

CA直连模式消除了临时握手页存在的理由：

1. **无需5分钟窗口** — CA证书永久有效（直到过期），身份可以随时验证
2. **无需第三方信道传公钥** — 公钥仅在确认接受后交换，且通过已建立的TLS信道
3. **无需指纹验证** — CA已经完成了身份认证，不需要用户肉眼比对指纹
4. **用户节点长期在线** — 不像Web服务可能临时下线，KirinNet节点是个人基础设施，
   始终在线监听。好友请求是一个API调用，不是打开一个临时网页

**实现路径：**

```
Alice → 解析 bob.kirinnet.org 的 KirinDNS TXT 记录 → 获取 Bob 的身份信息
Alice → 直连 Bob 节点 → TLS握手（CA证书）
Alice → POST https://bob.kirinnet.org:port/.kirinnet/friend/request
         Body: { sender_uuid, sender_domain, sender_nick, message, timestamp }
         // 不含公钥，仅声明意图
Bob   → 界面显示好友请求 → 接受
Bob   → POST https://alice.kirinnet.org:port/.kirinnet/friend/accept
         Body（加密）: { accepter_public_key, ... }
         // 此时才交换公钥
Alice → 解密 → 存入好友列表 → 返回自己的公钥
双方 → 好友关系建立 → 后续通信通过 WebSocket 长连接
```

这本质上是**把CA当作带外信道**——政府已经在线下验证了Bob的身份并签发了证书，
Alice不需要再重复这个验证过程。CA做的是一次性的"物理身份→域名"的绑定，
此后所有的信任建立都可以复用这个绑定。

### 2.1.2 临时握手界面（Ephemeral Handshake Interface）

适用于非CA用户之间无法直连的场景。双方没有CA证书，无法通过TLS自动验证身份，
需要一个临时端点来发起好友请求——但仍遵循两阶段模式。

**流程（同样两阶段：请求不含密钥，接受后交换）：**

```
阶段一：通过临时端点发起请求

1. Alice 生成一次性握手指令：
   - 生成随机 token (256位)
   - 创建临时端点: https://alice.kirinnet.org/.kirinnet/handshake/<token>
   - 有效期：默认 5 分钟
   - 该端点仅用于接收好友请求，不暴露Alice的公钥

2. Alice 将临时链接通过任意信道发给 Bob（微信、短信等）

3. Bob 在有效期内访问该链接 → 界面显示:
   ╔══════════════════════════════════════╗
   ║  KirinNet 好友请求                    ║
   ║  Alice (alice.kirinnet.org)          ║
   ║  请求添加你为好友                      ║
   ║  ⚠ 该用户无CA认证                      ║
   ║  建议通过其他渠道确认身份后再接受         ║
   ║  [ 发送好友请求 ]                      ║
   ║  有效期剩余: 3分27秒                   ║
   ╚══════════════════════════════════════╝

4. Bob 点击"发送好友请求" → 界面通过 WebSocket 或 POST 提交:
   {
     sender_uuid: <Bob的UUID>,
     sender_domain: "bob.kirinnet.org",
     sender_nick: "Bob",
     message: "...",
     timestamp: <ISO8601>,
   }
   // 不含公钥，仅声明意图

5. 握手指令自动销毁（一次性token，过期即失效）

阶段二：接受后交换公钥

6. Alice 收到 Bob 的好友请求 → 界面显示 → Alice 接受
7. Alice → POST https://bob.kirinnet.org:port/.kirinnet/friend/accept
   Body（加密）: { accepter_public_key, ... }
8. Bob → 解密 → 存入好友列表 → 返回Bob的公钥
9. 双方 → 好友关系建立 → 后续通过 WebSocket 通信
```

**与非CA直连的区别：**

| | CA直连（2.1.1） | 临时握手（2.1.2） |
|---|---|---|
| 请求发起方式 | Alice 直连 Bob 节点 | Alice 发临时链接，Bob 来访问 |
| 身份验证 | CA证书自动完成 | 人类判断（看域名和上下文） |
| 适用场景 | 双方有CA证书 | 至少一方无CA证书 |
| 安全性 | 机器验证，最高 | 人类验证，中等 |

**安全边界：**

1. **链接截获** — 微信等平台可以看到临时链接，但链接仅指向一个请求接收端点，
   不包含Alice的公钥。攻击者即使访问链接也只能发送一个好友请求（无害的意图声明）
2. **token被抢用** — 攻击者抢在Bob之前使用token发送请求 → Bob访问时看到
   "此链接已失效" → Bob通知Alice换信道重发
3. **身份伪造** — 攻击者可以搭建一个看起来像Alice握手页的界面。这是非CA模式的
   固有限制——没有CA就没有机器验证的身份。用户需要通过上下文判断：
   "Alice刚在微信上给我发了这个链接，应该真的是她"

这就是"妥协但可用"——方便性提高了攻击面，但攻击可检测且影响有限。

### 2.2 密钥轮换

当用户需要更换密钥时，新密钥**必须由旧密钥签名**：

```
轮换消息 = {
  new_key: <新RSA公钥>,
  reason: "voluntary" | "compromised" | "device_lost",
  timestamp: <ISO8601>,
}
旧密钥签名(序列化(轮换消息))
```

好友收到后：
1. 用存储的旧公钥验证签名 → 通过 → 确认是本人操作 → 更新公钥
2. 验证失败 → 拒绝更新 → 保留旧公钥 → 触发告警

**旧密钥丢失怎么办：** 如果节点被物理摧毁导致旧密钥完全丢失，则退回到物理层验证——
线下联系所有好友，当面重新交换密钥。

### 2.3 好友删除与撤销

删除好友时，发送由自己密钥签名的撤销消息：

```
撤销消息 = {
  action: "revoke_friend",
  target_id: <对方UUID>,
  timestamp: <ISO8601>,
}
发送方私钥签名(撤销消息)
```

好友收到后，立即将该用户从好友列表中移除，并清除存储的对方公钥。

### 2.4 信任传导

如果你信任Alice，Alice信任Bob，你可以选择信任Bob。过程：

1. Alice用她的私钥签名Bob的公钥和身份信息 → 生成"推荐令牌"
2. 你将推荐令牌转发给你的节点
3. 你的节点验证：Alice的签名是否有效 → Alice是否在你的好友列表中 → 如果都通过，
   接受Bob为好友（但信任级别标注为"间接"）

信任传导可以有深度限制（最多2跳），防止信任泛滥。


## 三、心跳信号与攻击预警系统

### 3.1 心跳机制

每个节点对其好友列表中的每个好友维护一个心跳计数器：

```
心跳状态 = {
  好友ID,
  最后心跳时间,
  平均心跳间隔,      // 基于最近7天计算
  RSA密钥哈希,       // 当前存储的公钥哈希
  状态: "online" | "away" | "offline" | "warning",
}
```

心跳消息格式：

```
{
  type: "heartbeat",
  sender: <UUID>,
  timestamp: <ISO8601>,
  rsa_key_hash: <SHA256(当前使用中的公钥)>,
  seq: <单调递增序号>,
}
发送方私钥签名(序列化(心跳消息))
```

- 心跳间隔：正常在线状态推荐 30 秒
- 连续 3 次未收到心跳 → 状态切换为 "away"
- 连续 30 次未收到心跳 → 状态切换为 "offline"
- 好友可以根据心跳状态判断对方是否在线（用于IM显示在线状态）

### 3.2 RSA大范围错误预警

这是 KirinNet 的分布式入侵检测核心。当一个好友的RSA密钥不匹配时，节点执行：

```
预警逻辑：
1. 收到消息 → 用存储的公钥验证 → 失败
2. 检查该好友的心跳记录 → 历史RSA密钥哈希是否一致
3. 如果历史上的哈希与当前存储的一致，但收到的消息使用不同密钥：
   a. 触发本地告警："好友 [昵称] 的RSA密钥异常"
   b. 向该好友的其他共同好友发送交叉验证请求（通过已加密信道）
   c. 如果共同好友也报告异常 → 升级为"可能被劫持"告警
   d. 通知用户："好友 [昵称] 可能被劫持，请线下确认"
```

关键点：攻击者无法同时伪造所有好友的公钥，因此这种交叉验证极难绕过。

### 3.3 流量骤降预警

与RSA错误预警配合使用：

```
流量骤降逻辑：
1. 监测每个好友的消息流入速率（消息数/小时）
2. 建立基准线（该好友过去7天的平均速率）
3. 如果速率突然下降超过80%且持续超过6小时：
   a. 检查RSA密钥是否也出现异常
   b. 如果两者同时触发 → 高概率劫持事件
   c. 如果仅流量下降 → 可能是该好友离线，不做告警升级
```

这个机制的精妙之处：攻击者如果接管了一个节点，他无法模仿原用户的行为模式
（跟谁聊天、频率如何），因此流量骤降成为另一个维度的异常检测信号。

### 3.4 预警信息互传

当节点检测到异常时，可以将预警信息加密广播给其他好友：

```
预警广播 = {
  type: "security_alert",
  subject: <被怀疑的好友UUID>,
  alert_type: "rsa_mismatch" | "traffic_drop" | "both",
  detected_at: <ISO8601>,
  evidence_hash: <SHA256(触发预警的消息)>,
}
检测方私钥签名(预警广播)
```

接收方本地交叉验证后，如果自己的检测结果与预警一致，加强告警级别。

**隐私保护：** 预警广播仅发送给被怀疑用户的共同好友（通过好友列表交叉计算）。
不会向全网广播，避免不必要的隐私泄露。


## 四、内置IM的安全保证

KirinNet的IM不同于任何已有的即时通讯工具，其安全保证体现在协议层面：

### 4.1 消息的端到端加密

```
发送方 → 用接收方公钥加密消息体 → 用自己的私钥签名整个包 → 发送
接收方 → 用发送方公钥验证签名 → 用自己的私钥解密消息体 → 读取
```

任何一步验证失败 → 消息被丢弃 → 触发3.2节描述的预警机制。

### 4.2 没有"平台"可以泄露消息

传统IM（无论是Signal还是微信）都有一个服务器中转消息。服务器可能在法律压力下
配合监控。KirinNet的消息直接在对等节点之间传输（或通过好友节点路由），不存在
一个中心化服务器持有所有用户的通信记录。

### 4.3 好友验证=身份验证

与传统IM不同，KirinNet不需要"手机号验证"或"邮箱验证"。你的身份就是你的域名，
你的域名对应的RSA密钥存储在你的所有好友的节点上。如果有人冒充你给某个好友发消息，
该好友本地的RSA公钥会立即发现签名不匹配，消息被拒绝。不需要Telegram那样的
"登录验证码"，不需要WhatsApp那样的"安全码变更通知"——一切都是自动的，因为好友
列表本身就是分布式身份账本。


## 五、需要完全匿名的用户

KirinNet不强制要求政府CA认证。一个用户可以选择：

- **不申请CA证书** → 其他人访问他的域名时，浏览器显示"未认证"
- **仅使用好友网络** → 好友之间通过RSA自验证，不依赖CA
- **获取好友的信任** → 通过线下见面交换公钥，建立信任

需要完全匿名的用户（如异议人士、记者、告密者）：
- 可以运行KirinNet节点而不申请CA证书
- 获得好友信任的方式完全离线（线下交换公钥）
- 好友之外的陌生人无法验证其身份，因此不会信任他
- 这是"各取所需"——需要匿名的放弃公共层信任，需要公共声誉的获取CA认证

**不存在中间状态：** 你不能既想被陌生人信任，又不想暴露物理身份。这正是物理世界的
逻辑——你可以在街头自称"张三"，但没有人会跟你签合同，除非你能出示身份证。


## 六、与传统威胁模型的对比

| 威胁 | 传统互联网 | KirinNet |
|------|-----------|----------|
| 网站数据库泄露 | 百万用户密码泄露 | 不存在，因为没有平台持有密码 |
| 撞库攻击 | 常见且灾难性 | 不存在，因为没有跨平台的身份概念 |
| CA被攻破 | 可为任意域名签发证书 | 范围限于该CA所属国家，且仅影响陌生人层 |
| 中间人攻击 | TLS握手阶段可能被劫持 | 好友层：公钥已锁定，无法劫持 |
| 服务器被入侵 | 所有用户数据暴露 | 仅单个用户节点暴露 |
| 政府强制数据调取 | 平台交出所有用户数据 | 不存在平台，数据分散在用户节点上 |
| 社交工程攻击 | 客服被欺骗重置密码 | 不存在客服，密钥由用户自己保管 |
| 设备丢失/被盗 | 所有在线账号面临风险 | 仅该节点数据暴露，无法影响好友 |


## 七、尚未解决的问题

### 7.1 垃圾信息和Sybil攻击

任何人都可以注册域名并申请CA证书。恶意用户可能创建大量节点发送垃圾好友请求。

可能的缓解措施：
- 好友请求附带少量PoW（工作量证明）
- 来自非好友的消息默认进入"请求"文件夹
- 好友关系需要双向确认

### 7.2 密钥存储安全

RSA私钥存储在用户节点上。如果节点是普通家用电脑，私钥安全性取决于该电脑的安全。

建议：私钥加密存储（用密码短语加密）。每次启动节点时需要输入密码解锁私钥。

### 7.3 好友网络的社交图谱隐私

好友列表本身是敏感信息。如果好友列表被明文存储，节点被物理攻破后，攻击者会获得该
用户的完整社交关系图。

建议：好友列表中的昵称和域名可以用用户自己的主密钥加密存储。

### 7.4 政府CA的跨境互认

目前只定义了"每个国家有独立CA"，但跨境互认机制（两个国家的CA之间如何相互承认）
尚未定义。这可能涉及外交层面的协议，超出了技术规范的范畴。


## 附录：与现有方案的对比

| 方案 | 身份模型 | 信任锚 | 密钥交换 | 中间人防御 |
|------|---------|--------|---------|-----------|
| Web PKI (TLS) | 域名 | CA | 每次TLS握手 | CA信任链 |
| PGP/GPG | 密钥指纹 | WoT/手动 | 线下交换或密钥服务器 | 信任网 |
| Signal | 手机号 | Signal服务器 | 首次通信 | 安全码比对 |
| Keybase | 社交账号 | 多个社交平台 | 平台证明 | 多平台交叉验证 |
| **KirinNet** | **域名** | **好友RSA + 政府CA** | **带外(QR/语音/介绍)** | **公钥锁定+分布式告警** |

KirinNet的独特之处：将"好友"本身作为分布式身份账本，同时用政府CA解决物理真实性。
它不假设任何第三方可信，但也不放弃与现实世界的身份关联。两个看似矛盾的目标（完全
自主 vs 物理可追溯）在KirinNet中不是互斥的——你选择暴露多少，就获得多少信任。
